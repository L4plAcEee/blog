前言

这一整个学习下来，主要是项目的基本介绍和环境配置，学习工程化的思想。
工程化是个好东西啊，真的方便项目的管理。开发不是难事，难的是持续开发。
持续开发，难免史上雕花，而工程化，就是尽可能的避免拉史。
然后就是跟着敲了TODO，就没啥了。

环境配置

前端环境配置

整个课程已经提供了Windows环境下的Nginx-1.20.2软件包，且其中已经包含了前端文件，
所以直接启动Nginx服务就行。
Nginx涉及配置也已经提供，所以无需担忧。
对于Nginx配置中涉及到的负载均衡等高级问题，这个学习阶段无需担忧。

  • 课程所用Nginx版本为1.20.2
  • 包含前端文件的意思是Nginx的html目录下已经打包好了静态页面文件

IDE：IDEA

IDEA直接从官网下了最新版，反正我的账号已经学生认证，不用白不用。

DB：MySQL

课程使用的是MySQL，但是没有具体提到版本。我用的是5.7。
因为事先接触过Docker技术，所以果断选择了wsl环境下拉了个mysql5.7的镜像，并挂载conf和课程提供的sql文件生成镜像。
后面调试的时候发现中文发生了乱码，因为以前遇见过类似错误，所以很快锁定到，是编码集的问题。
一查果然是，遂重新以utf-8的格式进行编码数据库。
这里偷了个懒，直接删了镜像重新创建了。

  • 发生这等错误的缘故好像是因为MySQL5.7默认编码集不是utf-8的问题

新概念

AOP（面向切面编程，Aspect-Oriented Programming）

定义：
AOP 是一种编程范式，旨在将横切关注点（如日志、事务管理、安全等）从核心业务逻辑中分离，以提升代码的模块化和可维护性。

核心概念：

  1. 切面（Aspect）：封装横切功能的模块（如日志工具类）。
  2. 连接点（Join Point）：程序执行中可插入切面的点（如方法调用、异常抛出）。
  3. 通知（Advice）：切面在连接点执行的具体逻辑（如前置日志、后置清理）。
  4. 切入点（Pointcut）：通过表达式定义哪些连接点会触发通知（如匹配所有 save* 方法）。

实现方式：

  • 静态织入（如 AspectJ）：编译期或类加载时修改字节码。
  • 动态代理（如 Spring AOP）：运行时生成代理对象，适用于接口或特定类。

应用场景：

  • 日志记录、性能监控
  • 事务管理、权限校验
  • 异常处理、缓存

优势：

  • 减少重复代码，业务逻辑更清晰。
  • 横切功能集中管理，易于维护。

反射（Reflection）

定义：
反射是程序在运行时动态获取类型信息（类、方法、字段等）并操作对象的能力，无需在编译时确定具体类型。

核心功能：

  1. 动态获取类型信息（如类名、父类、接口）。
  2. 创建对象（如通过无参/带参构造器实例化）。
  3. 调用方法（包括私有方法）。
  4. 访问/修改字段值（包括私有字段）。

实现示例（Java）：

应用场景：

  • 框架开发（如 Spring 的依赖注入）。
  • 序列化/反序列化（如 JSON 库解析私有字段）。
  • 动态代理、插件系统。

缺点：

  • 性能开销：运行时解析类型比静态调用慢。
  • 安全风险：可能绕过访问权限（需安全管理器控制）。

对比 AOP 与 反射

特性 AOP 反射
目的 解耦横切关注点 动态操作类型和对象
实现层次 代码结构设计（模块化） 语言特性（运行时类型操作）
典型应用 日志、事务 框架、动态代理、序列化
性能影响 依赖实现（动态代理或字节码织入） 显著（频繁反射调用需优化）

关联点：反射可用于实现 AOP（如动态代理基于反射调用目标方法）。

总结：

  • AOP 通过切面分离关注点，优化代码结构。
  • 反射 通过运行时动态能力，增强灵活性。
    两者结合能构建高扩展性系统（如 Spring 框架）。
