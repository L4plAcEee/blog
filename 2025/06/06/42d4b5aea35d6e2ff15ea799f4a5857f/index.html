前置知识

HTML

什么是HTML？

HTML 是用来描述网页的一种语言。

  • HTML 指的是超文本标记语言: HyperText Markup Language
  • HTML 不是一种编程语言，而是一种标记语言
  • 标记语言是一套标记标签 (markup tag)
  • HTML 使用标记标签来描述网页
  • HTML 文档包含了HTML 标签及文本内容
  • HTML文档也叫做 web 页面

HTML标签与元素

HTML 标记标签通常被称为 HTML 标签 (HTML tag)。

  • HTML 标签是由_尖括号_包围的关键词，比如 <html>
  • HTML 标签通常是_成对出现_的，比如 <b> 和 </b>
  • 标签对中的第一个标签是 _开始标签_，第二个标签是 结束标签
  • 开始和结束标签也被称为 开放标签 和 闭合标签

“HTML 标签” 和 “HTML 元素” 通常都是描述同样的意思.
但是严格来讲, 一个 HTML 元素包含了开始标签与结束标签，如下实例:
HTML 元素:
<p>这是一个段落。</p>

基本结构

  • <!DOCTYPE html> 声明为 HTML5 文档
  • <html> 元素是 HTML 页面的根元素
  • <head> 元素包含了文档的元（meta）数据，如 <meta charset="utf-8"> 定义网页编码格式为 utf-8。
  • <title> 元素描述了文档的标题
  • <body> 元素包含了可见的页面内容
  • <h1> 元素定义一个大标题
  • <p> 元素定义一个段落

基本标签

  • HTML 标题（Heading）是通过<h1> - <h6> 标签来定义的。
  • HTML 段落是通过标签 <p> 来定义的。
  • HTML 链接是通过标签 <a> 来定义的。
  • HTML 图像是通过标签 <img> 来定义的.
  •  <p> 元素定义了 HTML 文档中的一个段落。这个元素拥有一个开始标签 <p> 以及一个结束标签 </p>.
  • <body> 元素定义了 HTML 文档的主体。 这个元素拥有一个开始标签 <body> 以及一个结束标签 </body>。
  • <html> 元素定义了整个 HTML 文档。 这个元素拥有一个开始标签 <html> ，以及一个结束标签 </html>.

注意事项

  • 目前在大部分浏览器中，直接输出中文会出现中文乱码的情况，这时候我们就需要在头部将字符声明为 UTF-8 或 GBK。
  • 即使您忘记了使用结束标签，大多数浏览器也会正确地显示 HTML, 但不要依赖这种做法。忘记使用结束标签会产生不可预料的结果或错误。

HTTP协议

HTTP（超文本传输协议，HyperText Transfer Protocol）是一种用于分布式、协作式、超媒体信息系统的应用层协议。

HTTP 是万维网（WWW）的数据通信的基础，设计目的是确保客户端与服务器之间的通信，是互联网上最常用的协议之一。

HTTP 是一个基于 TCP/IP 通信协议来传递数据的（HTML 文件、图片文件、查询结果等）。

设计 HTTP 最初的目的是为了提供一种发布和接收 HTML 页面的方法，通过 HTTP 或者 HTTPS 协议请求的资源由统一资源标识符（Uniform Resource Identifiers，URI）来标识。

HTTP 的请求-响应

HTTP 的基本工作原理是客户端（通常是 web 浏览器）向服务器发送请求，服务器接收到请求后，返回相应的资源。这些资源可以是网页、图像、音频文件、视频等。

HTTP 使用了客户端-服务器模型，其中客户端发送请求，服务器返回响应。

HTTP 的请求-响应模型通常由以下几个步骤组成：

  • 建立连接：客户端与服务器之间建立连接。在传统的 HTTP 中，这是基于 TCP/IP 协议的。最近的 HTTP/2 和 HTTP/3 则使用了更先进的传输层协议，例如基于 TCP 的二进制协议（HTTP/2）或基于 UDP 的 QUIC 协议（HTTP/3）。

  • 发送请求：客户端向服务器发送请求，请求中包含要访问的资源的 URL、请求方法（GET、POST、PUT、DELETE 等）、请求头（例如，Accept、User-Agent）以及可选的请求体（对于 POST 或 PUT 请求）。

  • 处理请求：服务器接收到请求后，根据请求中的信息找到相应的资源，执行相应的处理操作。这可能涉及从数据库中检索数据、生成动态内容或者简单地返回静态文件。

  • 发送响应：服务器将处理后的结果封装在响应中，并将其发送回客户端。响应包含状态码（用于指示请求的成功或失败）、响应头（例如，Content-Type、Content-Length）以及可选的响应体（例如，HTML 页面、图像数据）。

  • 关闭连接：在完成请求-响应周期后，客户端和服务器之间的连接可以被关闭，除非使用了持久连接（如 HTTP/1.1 中的 keep-alive）。

HTTP 方法

HTTP 方法指定了客户端可以对服务器上的资源执行哪些动作。

主要的HTTP方法有：

  • GET：请求从服务器获取指定资源。这是最常用的方法，用于访问页面。
  • POST：请求服务器接受并处理请求体中的数据，通常用于表单提交。
  • PUT：请求服务器存储一个资源，并用请求体中的内容替换目标资源的所有内容。
  • DELETE：请求服务器删除指定的资源。
  • HEAD：与 GET 类似，但不获取资源的内容，只获取响应头信息。

HTTP 状态码

HTTP状态码是服务器对客户端请求的响应。

状态码分为五类：

  • 1xx（信息性状态码）：表示接收的请求正在处理。
  • 2xx（成功状态码）：表示请求正常处理完毕。
  • 3xx（重定向状态码）：需要后续操作才能完成这一请求。
  • 4xx（客户端错误状态码）：表示请求包含语法错误或无法完成。
  • 5xx（服务器错误状态码）：服务器在处理请求的过程中发生了错误。

HTTP 版本

HTTP 有多个版本，目前广泛使用的是 HTTP/1.1 和 HTTP/2，以及正在逐步推广的 HTTP/3。

  • HTTP/1.1：支持持久连接，允许多个请求/响应通过同一个 TCP 连接传输，减少了建立和关闭连接的消耗。
  • HTTP/2：基于二进制分帧，支持多路复用，允许同时通过单一的 HTTP/2 连接发起多重的、独立的、双向的交流。
  • HTTP/3：基于 QUIC 协议，旨在减少网络延迟，提高传输速度和安全性。

请求与响应接口

  • HttpServletRequest 和 HttpServletResponse 是 HTTP 协议专用 的接口。

  • 它们作为参数传入 HttpServlet.service()，最终传给 doGet() / doPost() 等方法中使用。

  • 它们封装了 HTTP 协议中的各种信息（如请求头、响应状态码、重定向等）。

  • 表示客户端发来的 HTTP 请求。

  • 继承自 ServletRequest 接口。

  • 提供了如 getParameter()、getHeader()、getCookies()、getSession() 等方法。

HttpServletResponse 接口

  • 表示发送给客户端的 HTTP 响应。

  • 继承自 ServletResponse 接口。

  • 提供了如 setStatus()、setHeader()、getWriter()、sendRedirect() 等方法。

会话技术

Cookie 用于标识用户身份，Session 用于保存用户的会话上下文信息。

  • Cookie：保存在客户端，通常用来存储 SessionID 或 Token。

  • Session：保存在服务器端，根据 Cookie 提供的标识（如 JSESSIONID）来查找对应的会话数据。

过滤器和监听器

过滤器（Filter）拦截请求与响应，监听器（Listener）监听 Web 应用中的生命周期事件。

概念解析

1. URL（统一资源定位符）

  • 定义与结构：

    • 格式：<协议>://<主机名>:<端口>/<路径>?<查询参数>

    • 示例：http://localhost:8080/myapp/index.jsp?user=alice

  • 作用：唯一标识网络资源的位置。

  • 常见要点：

    1. 协议（Protocol）：如 HTTP/HTTPS；

    2. 主机名（Host）：域名或 IP；

    3. 端口（Port）：HTTP 默认 80，HTTPS 默认 443；

    4. 路径（Path）：服务器上资源的相对地址；

    5. 查询参数（Query String）：? 后面的键值对，可由 & 分隔多个参数。

2. 虚拟目录（Context Path）

  • 概念：

    • 在 Java Web 容器中，通过 <项目名称> 或在部署时指定的名称来映射应用访问路径。

    • 访问示例：部署为 myapp 时，URL 为 http://localhost:8080/myapp/。

  • 获取方式：

  • 用途：

    • 用于组织多个 Web 应用共存；

    • 在 JSP/Servlet 中动态拼接资源路径时，常用 ${pageContext.request.contextPath} 或 request.getContextPath() 保证可移植性。

3. RequestDispatcher

  • 作用：在服务器端进行资源转发（Forward）或包含（Include）。

  • 获取方式：

  • 主要方法：

    1. rd.forward(request, response)——从当前 Servlet/Servlet 输出流“转发”到目标资源，客户端地址栏 URL 不变；

    2. rd.include(request, response)——在响应中嵌入目标资源输出，常用于公共页头、页尾包含。

  • 应用场景：

    • Servlet 处理逻辑完成后，将请求转给 JSP 渲染；

    • 多个 Servlet/Filter 之间共享请求；

    • 页面局部重用：公共菜单、页眉/页脚。

4. Tomcat

  • 简介：

    • Apache Tomcat 是开源的 Java Web 服务器与 Servlet 容器，实现了 Servlet、JSP 规范。

    • 常见版本：Tomcat 8、Tomcat 9、Tomcat 10 等；不同版本对应不同的 Servlet/JSP 规范。

  • 主要功能：

    1. 生命周期管理：启动/停止 Web 应用，加载 web.xml、读取注解、初始化 ServletContext；

    2. 请求分发：将客户端 HTTP 请求映射到对应的 Servlet/JSP；

    3. Session 管理：生成并维护 JSESSIONID，支持持久化（可配置）；

    4. 虚拟主机：通过 <Host> 标签支持多个域名或项目共存；

    5. 安全控制：支持基本的 Security Realm、用户/角色管理。

  • 常见配置点：

    • <server.xml> 中的 <Connector>（端口、编码、线程池）；

    • <context.xml> 或项目的 META-INF/context.xml（资源映射、会话配置）；

    • webapps/ 目录：将 Web 应用直接解压到此目录即可自动部署。

5. Session 技术

  • 概念：

    • Session 是服务器端保存用户会话状态的数据结构，每个会话对应唯一的 Session ID；

    • Session 数据一般存放在服务器内存、文件或外部持久化存储（如 Redis）。

  • 生命周期：

    1. 创建：调用 request.getSession()（若不存在，则新建）；

    2. 访问：通过 session.setAttribute(String name, Object value) 存储，session.getAttribute(String name) 读取；

    3. 失效：调用 session.invalidate() 显式销毁，或达到超时时间（默认 30 分钟）后自动失效；

  • 特点：

    • 服务器保存：相比 Cookie，数据存放在服务器，更安全；

    • 依赖 Cookie/URL 重写：Session ID 通过 Cookie（JSESSIONID），若禁用 Cookie 可通过 URL 重写 jsessionid=xxxx 传播；

    • 资源消耗：每个活跃用户会话都占用服务器资源；大并发需考虑分布式 Session、共享存储或无状态设计（如 JWT）。

  • 常见应用：

    • 用户登录鉴权：登录后在 Session 中存储用户信息；

    • 购物车：将临时购物数据存储在 Session；

    • 临时缓存：短期保存页面跳转间需要共享的数据。

8. HTTP/1.1 协议的通信过程

  1. 建立 TCP 连接（Three-way Handshake）

    • 客户端向服务器发送 SYN，服务器回 SYN-ACK，客户端回 ACK，连接建立。

  2. 发送 HTTP 请求报文

    • 请求行：如 GET /index.html HTTP/1.1；

    • 请求头：Host: www.example.com、User-Agent: …、Accept: … 等；

    • 可选请求体：如 POST 方法提交的表单数据、JSON 等。

  3. 服务器处理并返回响应报文

    • 状态行：如 HTTP/1.1 200 OK；

    • 响应头：Content-Type: text/html;charset=UTF-8、Content-Length: ... 等；

    • 响应体：实际内容（HTML、JSON、图片字节流等）。

  4. 客户端解析并渲染

    • 浏览器根据状态码、响应头判断资源类型并处理：

      • 若为 HTML，解析后请求相关资源（CSS、JS、图片）；

      • 若为文件下载，弹出“另存为”。

  5. 连接管理

    • HTTP/1.1 默认启用持久连接（Connection: keep-alive），可在多个请求之间复用同一 TCP 连接；

    • 若加上 Connection: close，则本次响应完成后立即关闭连接。

  6. 管道化与分块传输（可选）

    • 管道化（Pipelining）：客户端在接收到第一个响应之前可以并行发送多个请求（浏览器普遍未启用）；

    • 分块传输编码（Chunked）：在响应头中设置 Transfer-Encoding: chunked，服务器边生成边发送数据片段，适合大文件或动态内容。

9. Cookie 的运行机制

  • 定义：Cookie 是服务器通过 HTTP 响应头 Set-Cookie 下发给浏览器的小段文本，用于在同一域名下保存用户相关信息。

  • 关键流程：

    1. 下发：服务器返回响应时，在响应头中携带 Set-Cookie: name=value; Domain=...; Path=...; Expires=...; HttpOnly; Secure；

    2. 存储：浏览器将 Cookie 保存在本地，存储位置与浏览器类型有关；

    3. 自动携带：之后对同一域名的请求，浏览器自动在请求头中添加 Cookie: name=value; ...；

    4. 服务端解析：Servlet 通过 request.getCookies() 或 request.getHeader("Cookie") 获取并解析。

  • 属性说明：

    • Domain：限定可访问此 Cookie 的域；

    • Path：限定可访问路径；

    • Expires/Max-Age：过期时间，设置后为持久化 Cookie，否则为会话级（浏览器关闭后失效）；

    • HttpOnly：标记后前端 JS 无法读取，防止 XSS 攻击；

    • Secure：仅在 HTTPS 连接中发送。

  • 应用场景：

    • 保存登录状态（Session ID）；

    • 记录用户偏好设置；

    • 简单的跟踪与分析。

10. 会话（Session）的概念和特点

此处补充第 5 节中关于 Session 基本概念，因与“Cookie 机制”紧密相关，集中说明如下：

  • 创建与识别：调用 HttpSession session = request.getSession()；

  • 服务器存储：服务端维护一个 Map<String,Object> 存储会话数据，客户端仅持有 JSESSIONID；

  • 生命周期：

    1. 创建：首次调用 getSession(true) 时创建；

    2. 访问更新：对 Session 有操作或重新访问时，重置超时时间；

    3. 失效：超时（默认 30min）或调用 session.invalidate()；

  • 安全性：

    • 客户端只能看到 JSESSIONID，无法篡改服务端数据；

    • 需防范 Session 劫持：可通过开启 HTTPS、设置 HttpOnly Cookie、使用防御性过期策略等。

  • 分布式场景：

    • 单机部署：Session 存储在内存；

    • 分布式部署：可使用 Session 粘滞（同一用户固定到同一台服务器）或外部共享存储（如 Redis、数据库）。

15.4. EL（Expression Language）语言

  • 基本语法：

  • 引入方式（在 JSP 顶部）：

  • 主要标签库类别与常用示例：

    1. Core 核心标签库（c）

      • <c:if test="${not empty list}">…</c:if>：条件判断；

      • <c:choose> <c:when test="…">…</c:when> <c:otherwise>…</c:otherwise> </c:choose>：多分支判断；

  • 典型应用场景：

    • 视图层完全不使用脚本式 Java 代码，将业务逻辑控制权交给 Servlet/Controller，JSP 专注展示；

    • 结合 Spring MVC、Struts2 等框架时，JSTL 与 EL 是前端页面标签化访问数据的基础；

    • 国际化、多语言页面开发；

    • 列表渲染、分页组件、表单校验提示等。

  1. B/S架构（Browser/Server架构）：

  • 使用PrintWriter输出当前系统时间。
    功能上类似一个动态时钟页面，自动更新显示时间。

  1. 代码功能简述：

该Servlet实现了一个简易的购物车功能。

  • 通过HttpSession获取或创建用户的购物车（List）。

  • 如果购物车不存在，则创建新的购物车对象并绑定到Session。

  • 创建一个名为“婺源酒糟鱼”的商品对象，加入购物车。

  • 通过Cookie保存Session ID，设置有效期为1小时，路径限制为/ch7。
    作用是实现用户的会话购物车，用户访问时商品会加入购物车，并通过Cookie维护会话状态。

  1. 程序功能分析：

这是一个JSP页面，功能是显示当前时间并每秒自动刷新页面。

  • <meta http-equiv="refresh" content="1"> 实现页面每秒刷新一次。

  • JSP表达式<%= new java.util.Date().toLocaleString() %>输出当前时间字符串。
    页面类似一个动态时钟，持续更新显示当前时间。
